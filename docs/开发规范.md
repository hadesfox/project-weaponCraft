# AI 代码工程规范 (Engineering Principles)

## 0. 最高原则

本规范定义代码必须遵守的工程质量标准，优先级高于一切便捷性考量。在编写或修改任何代码前，请确认已理解以下原则。所有产出代码都必须可被审查、可被复用、可被独立修改。

---

## 1. 高内聚 (High Cohesion)

- **单一职责**：每个函数 / 模块 / 类只做一件事。如果一个函数的名字需要用"和""并且"来描述，就必须拆分。
- **逻辑聚合**：相关的数据和操作放在一起，不相关的逻辑禁止塞进同一个函数。
- **函数粒度**：单个函数原则上不超过 50 行，超过则考虑拆分为更小的辅助函数。
- **就近原则**：只在一个地方使用的逻辑，不要过度抽象成全局工具；多处复用的逻辑，才提取为公共模块。

---

## 2. 低耦合 (Low Coupling)

- **依赖明确**：模块之间通过明确的参数 / 接口通信，禁止隐式依赖全局状态。
- **禁止散落的全局变量**：状态必须集中管理，不允许在多个地方随意读写同一个全局变量。
- **面向接口而非实现**：上层逻辑只依赖抽象的方法签名，不依赖底层的具体实现细节。
- **单向依赖**：避免模块之间循环引用（A 依赖 B，B 又依赖 A）。数据流向应尽量清晰、单向。
- **隔离副作用**：把"计算逻辑"（纯函数）和"副作用"（读写文件、修改全局状态、渲染输出）分离开。

---

## 3. 可读性 (Readability)

- **命名自解释**：变量 / 函数命名要表达意图，禁止使用 a、b、tmp、data1 这类无意义命名。
- **消除魔法数字**：禁止在逻辑中直接出现裸数字 / 裸字符串，必须定义为有名字的常量。
- **注释讲"为什么"**：注释解释意图和原因，而不是复述代码在做什么。
- **统一风格**：缩进、命名风格（驼峰 / 下划线）、括号位置在整个项目中保持一致。

---

## 4. 可维护性 (Maintainability)

- **DRY 原则**：杜绝复制粘贴。重复出现两次以上的逻辑必须抽取复用。
- **配置与逻辑分离**：可变参数集中在配置区，调整行为时不需要改动核心逻辑代码。
- **避免深层嵌套**：嵌套层级不超过 3 层，优先使用"提前返回（early return）"来减少嵌套。
- **错误处理显式化**：对可能失败的操作要有明确的错误处理，不允许静默吞掉异常。

---

## 5. 可扩展性 (Extensibility)

- **开闭原则**：增加新功能时应通过"新增"代码实现，而不是大改现有逻辑。
- **预留扩展点**：对未来可能变化的部分（如规则、数值、类型），用数据驱动 / 配置驱动的方式设计，便于扩展。
- **避免硬编码分支膨胀**：当 if/else 或 switch 分支不断增长时，改用映射表（map / 字典）或多态结构。

---

## 6. 交付与协作约束

- **改动最小化**：修复或新增功能时，不擅自改动无关的、已正常工作的代码。
- **解释设计决策**：当采用某种结构（如拆分模块、引入抽象）时，用一两句话说明这样设计的内聚 / 解耦理由。
- **重构需告知**：若发现现有代码违反上述原则，可提出重构建议，但必须先说明问题，经我确认后再动手。
- **拒绝过度设计**：在满足上述原则的前提下保持简洁，不为了"可能的未来"引入无必要的抽象层（避免过度工程 YAGNI）。

---

## 7. 代码格式规范 (Code Formatting)

- **缩进**：使用 4 个空格（禁止制表符）
- **换行**：每行不超过 120 字符
- **空行**：函数之间空 2 行，逻辑块之间空 1 行，文件末尾保留 1 个空行
- **括号**：采用 K&R 风格（左括号不换行）
- **运算符**：二元运算符前后各留 1 个空格
- **逗号**：逗号后必须跟空格
- **空格**：关键字后、括号前留空格（`if (x)` 而非 `if(x)`）
- **文件长度**：单个源文件不超过 1000 行，超过则考虑按职责拆分为多个文件

---

## 8. 命名规范 (Naming Conventions)

### 8.1 基础规则
- **变量**：小驼峰命名 `playerHealth`、`damageAmount`
- **函数**：小驼峰命名 `calculateDamage()`、`loadWeaponData()`
- **类/模块**：大驼峰命名 `PlayerController`、`WeaponSystem`
- **常量**：全大写 + 下划线 `MAX_HEALTH`、`BASE_DAMAGE`
- **私有成员**：下划线前缀 `_privateVar`、`_internalState`

### 8.2 文件命名
- **脚本文件**：下划线分隔 `player_controller.lua`、`weapon_data.json`
- **目录结构**：按功能模块组织，使用小写字母

### 8.3 特殊约定
- **布尔变量**：使用 `is`/`has`/`can` 前缀 `isActive`、`hasWeapon`
- **集合变量**：使用复数形式 `enemies`、`weapons`
- **临时变量**：禁止使用 `tmp`/`temp`，改用描述性名称

---

## 9. 注释规范 (Comment Standards)

### 9.1 文档注释（LDoc/JSDoc 风格）
```lua
--- 计算武器伤害值
---@param attack number 攻击力（范围：1-100）
---@param defense number 防御力（范围：0-50）
---@param isCritical boolean 是否暴击
---@return number 最终伤害值
function calculateDamage(attack, defense, isCritical)
```

### 9.2 行内注释
- 复杂逻辑必须有注释说明意图
- 禁止注释复述代码（如 `-- 加 1`）
- 注释与代码之间空 1 个空格

### 9.3 待办事项标记
- `TODO`: 需要实现的功能
- `FIXME`: 需要修复的问题
- `NOTE`: 重要说明
- 必须标注处理人：`-- TODO: 优化性能 [张三]`

---

## 10. Git 版本控制规范 (Version Control)

### 10.1 分支策略
- `main`：主分支，保持稳定可发布状态
- `develop`：开发分支，整合所有功能
- `feature/xxx`：功能分支，从 develop 创建，完成后合并回 develop
- `hotfix/xxx`：紧急修复分支，从 main 创建，修复后同时合并到 main 和 develop

### 10.2 Commit 消息规范
```
type(scope): brief description (max 50 chars)

Optional detailed description (wrap at 72 chars)
```

**类型说明**：
- `feat`: 新增功能
- `fix`: 修复 Bug
- `docs`: 文档更新
- `refactor`: 代码重构（无功能变更）
- `test`: 测试相关
- `chore`: 构建/工具配置

**示例**：
```
feat(weapon): 添加武器类型识别
fix(forge): 修复淬火提示文字遮挡问题
docs: 更新开发规范文档
```

### 10.3 PR/MR 规范
- 每个 PR 只解决一个问题或实现一个功能
- PR 描述必须包含：变更目的、改动内容、测试方式
- 必须通过 CI 检查
- 至少需要 1 人审查通过

---

## 11. 测试规范 (Testing Standards)

### 11.1 测试覆盖率
- 单元测试覆盖率 ≥ 70%
- 核心逻辑必须有测试覆盖

### 11.2 测试文件组织
- 测试文件与源文件同名，后缀 `_test.lua`
- 测试函数命名：`Test_Module_Function_Scenario`

### 11.3 测试内容要求
- **单元测试**：测试单个函数/模块的独立行为
- **集成测试**：测试多个模块协同工作
- **边界测试**：测试输入边界值、空值、异常情况

### 11.4 测试数据
- 使用明确的测试数据，禁止使用魔术数字
- 测试数据应覆盖正常、边界、异常三种情况

---

## 12. 错误处理规范 (Error Handling)

### 12.1 异常捕获
- 对可能失败的操作必须有 try-catch 或错误检查
- 禁止静默吞掉异常（`catch { }` 空块）

### 12.2 错误日志
- 使用统一的日志工具记录错误
- 日志必须包含：时间、级别、模块、详细信息
- 敏感信息必须脱敏（密码、Token 等）

### 12.3 错误码定义
- 使用枚举或常量定义错误码
- 错误码必须有对应的错误信息说明

### 12.4 用户友好提示
- 用户看到的错误信息必须清晰易懂
- 技术细节只记录在日志中，不展示给用户

---

## 13. AI 编码高频隐患自查清单 (AI Code Pitfalls Self-Check)

由于这是黑盒开发——我无法看到代码运行过程，AI 也无法验证实际效果，因此你必须在每次产出代码后，主动对照以下清单逐项自查，并在回复末尾确认"已自查"。

### 13.1 幻觉与虚构 (Hallucination)

- 禁止虚构不存在的 API、函数、库、参数或配置项。所有调用的接口必须是真实存在且确实可用的。
- 不确定某个 API/方法是否存在或签名是否正确时，必须明确标注"此处不确定，请核实"，禁止假装确定。
- 禁止编造文件名、路径、字段名。引用任何变量/函数前，确认它在当前代码里确实已定义。

### 13.2 上下文断裂 (Context Loss)

- 修改代码前，先确认你引用的变量名、函数名、数据结构与"现有代码"完全一致，禁止凭印象写出不匹配的名字。
- 禁止偷偷修改函数的参数顺序、返回值结构、数据格式，若必须改动，要显式指出并同步所有调用处。
- 新增代码必须与现有命名风格、数据结构保持一致，不得另起一套互不兼容的写法。

### 13.3 边界与异常 (Edge Cases)

- 必须处理空值 / null / undefined / 空数组 / 空字符串的情况。
- 必须考虑边界值：0、负数、最大值、第一个元素、最后一个元素、列表为空时。
- 涉及数组索引、循环时，检查是否会越界（off-by-one 错误）。
- 涉及除法、取余时，检查除数是否可能为 0。
- 涉及异步 / 回调 / 定时器时，检查执行顺序和竞态条件，确保依赖的数据已就绪。

### 13.4 状态与副作用 (State & Side Effects)

- 检查是否存在变量未初始化就被使用的情况。
- 检查是否有变量被多处修改导致状态不可控（共享可变状态）。
- 检查循环 / 事件监听 / 定时器是否有重复注册、未清理而导致的泄漏或重复触发。
- 涉及引用类型（对象/数组）时，明确是值拷贝还是引用，避免误改原数据。

### 13.5 逻辑完整性 (Logic Completeness)

- 条件分支必须覆盖所有情况，if/switch 要考虑 else / default 的兜底。
- 提到的每个功能都要真正实现，禁止留下空函数、TODO 占位、假装实现但内部为空。
- 禁止"看起来对但实际没接线"的代码：新写的函数必须真正被调用，新加的配置必须真正生效。

### 13.6 一致性与残留 (Consistency)

- 修改后检查：是否有旧逻辑残留、是否有重复定义、是否有已废弃但仍被调用的代码。
- 增删字段/参数后，确认所有相关位置都已同步更新，不留"改了一半"的状态。
- 确认没有遗留调试用的临时代码影响正式逻辑（调试输出可保留，但临时 hack 必须清理）。

### 13.7 自查回执 (Mandatory)

每次给出代码后，必须在末尾附上简短自查结论，格式如下：

- 是否引用了未确认存在的 API：是/否
- 是否处理了空值与边界：是/否
- 新增逻辑是否已被实际调用：是/否
- 与现有代码命名/结构是否一致：是/否
- 存疑或需我确认的点：（列出，没有则写"无"）

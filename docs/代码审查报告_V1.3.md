## 二、问题清单

### 🔴 严重（违反规范硬性规则）

#### 问题 1: 多个函数远超 50 行限制

**违反规范**: §1 高内聚 — "单个函数原则上不超过50行"

| 文件 | 函数 | 行数 | 超出 |
|------|------|------|------|
| `ForgeState.lua` | `RenderHammerPhase()` | ~258 | 5x 超标 |
| `ForgeState.lua` | `RenderQuenchPhase()` | ~135 | 2.7x 超标 |
| `TrialState.lua` | `RenderPlayer()` | ~122 | 2.4x 超标 |
| `TrialState.lua` | `RenderAttack()` | ~99 | 2x 超标 |

**建议**: 按视觉区域拆分渲染函数，例如 `RenderHammerPhase` 可拆为:
- `RenderHammerBar()` — 进度条/节奏条
- `RenderHammerAnvil()` — 铁砧动画
- `RenderHammerFeedback()` — 命中反馈文字
- `RenderHammerScore()` — 分数显示

---

#### 问题 2: Config.lua 存在死代码

**违反规范**: §7.6 一致性与残留 — "是否有已废弃但仍被调用的代码"

以下三个配置项在 Config.lua 中定义，但整个项目中**从未被引用**：

```lua
-- Config.lua:46-48 — 从未被其他文件使用
HammerBeatInterval = 0.8,  -- 锤击节拍间隔（死代码）
PerfectWindow = 0.15,      -- 完美判定窗口（死代码）
GoodWindow = 0.3,          -- 良好判定窗口（死代码）
```

**原因**: ForgeState.lua 已改用随机区域判定系统（`PERFECT_HALF`, `GOOD_HALF` 本地常量），旧的时间窗口配置失去意义。

**建议**: 删除这三行死代码。

---

### 🟡 中等（违反规范建议性规则）

#### 问题 3: ForgeState 判定参数与 Config 脱节

**违反规范**: §4 可维护性 — "配置与逻辑分离：可变参数集中在配置区"

`ForgeState.lua` 定义了锤击判定的核心参数作为局部常量：

```lua
-- ForgeState.lua 顶部
local PERFECT_HALF = 0.10    -- 完美区域半宽
local GOOD_HALF = 0.40       -- 良好区域半宽
local ZONE_MARGIN = 0.40     -- 区域随机范围边距
```

这些是**游戏性核心参数**（直接影响难度），应纳入 `Config.Forge` 统一管理。当前 Config.Forge 保留了旧的 `PerfectWindow`/`GoodWindow` 却不生效，而实际生效的参数藏在逻辑代码中。

**建议**: 
1. 删除 Config.Forge 中的 `PerfectWindow`、`GoodWindow`、`HammerBeatInterval`
2. 新增: `PerfectHalf = 0.10`、`GoodHalf = 0.40`、`ZoneMargin = 0.40`
3. ForgeState 从 Config 读取这些值

---

#### 问题 4: TrialState.lua 超过 1500 行

**违反规范**: §1 高内聚（隐含规则：文件不宜过长）

`TrialState.lua` 已达 1627 行，是项目最大文件。虽已提取了 `Trial/Slime.lua`（274行），但主文件仍然臃肿。

**建议**: 考虑进一步提取：
- `Trial/Enemies.lua` — 敌人生成/AI/渲染
- `Trial/Platforms.lua` — 平台生成/渲染
- `Trial/Combat.lua` — 攻击/碰撞检测

### 🟢 轻微（优化建议）

#### 问题 5: 部分渲染代码存在魔法数字

**违反规范**: §3 可读性 — "消除魔法数字"

个别 NanoVG 颜色值直接写在逻辑中，未使用 `Config.Colors`：

```lua
-- ForgeState.lua 中 Perfect 反馈使用内联颜色
nvgFillColor(vg, nvgRGBA(80, 220, 100, 255))  -- 应为 Config.Colors.Success
nvgFillColor(vg, nvgRGBA(240, 80, 80, 255))   -- 应为 Config.Colors.Danger
```

**建议**: 对频繁使用的 UI 颜色，统一通过 `Config.Colors` 引用。

## 三、优点（值得保持）

1. **模块化架构清晰**: 状态机模式 + 独立模块（Drawing/Canvas、Trial/Slime）职责分明
2. **Config 集中管理**: 大部分可调参数都集中在 Config.lua，调试方便
3. **命名风格统一**: 全局采用 `camelCase_` 后缀标识模块私有变量
4. **main.lua 结构优秀**: 事件分发/状态切换逻辑清晰，302行控制在合理范围
